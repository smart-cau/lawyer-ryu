# 개발 시스템 (development)

이 디렉토리는 **개발 진행을 어떻게 운영하는지**에 대한 메타 문서다. 콘텐츠 SoT는 다른 `docs/` 디렉토리에, task 진행 추적은 GitHub에 있다.

## Task 추적은 GitHub에서

- **Issues**: https://github.com/smart-cau/lawyer-ryu/issues
- **Project 보드**: `lawyer-ryu v1` (user project — 최초 1회 권한 갱신 필요, [후속 작업](#후속-작업) 참고)

영역별 epic 4개(#1~#4) + sub-task 9개(#5~#13)로 분해됨. 라벨로 영역(`area:*`)·타입(`epic`/`planning`)·차단(`blocked:*`) 표시.

진행률·다음 액션을 이 디렉토리에 *복제하지 않는다* — 그건 Issue/Project가 단일 진실 원천. 여기는 **결정 이력**과 **산출물 매핑 룰**을 다룬다.

## 산출물은 어디로 가는가

### 기존 `docs/` 디렉토리의 성격

먼저 기존 `docs/`가 어떤 문서들로 짜여 있는지 짚어둔다. **출처별 자료조사 + 일부 가공물(IA·스키마) + 메타** 구조이며, *페이지 구조와 1:1이 아니다*.

| 디렉토리              | 성격                                  | 페이지와 관계                                  |
| ----------------- | ----------------------------------- | ---------------------------------------- |
| `01-profile/`     | 변호사 프로필 자료조사 (학력·경력·인적사항)           | about/lawyer 콘텐츠 SoT                     |
| `02-services/`    | 업무분야 IA·1차 청취 원문 (자료조사 + 가공물 혼재)    | services 콘텐츠·라우팅 SoT (영역 1:1)            |
| `03-firm/`        | 법무법인 인유 정보·USP·브랜드 포지셔닝             | main(랜딩)·about/firm(미래) 콘텐츠 SoT          |
| `04-cases/`       | 언론기사 메타스키마·수집 데이터 (자료조사 + 가공물 혼재)   | cases 콘텐츠 SoT                            |
| `05-references/`  | 벤치마킹·기획 참고자료                        | 페이지 무관, 영구 자산                            |
| `06-interview/`   | 변호사 인터뷰 준비·진행·결과                    | about 콘텐츠·전 영역 결정 입력                     |
| `07-development/` | 개발 시스템 메타 (이 디렉토리)                  | 페이지 무관, 메타                               |
| `assets/`         | 이미지·로고·원본 에셋                        | 페이지 무관, 영구 에셋                            |

한 페이지가 여러 디렉토리를 인용하는 게 자연이다 (예: 랜딩 = `03-firm` + `01-profile` + `02-services` 모두 인용). 페이지 단위로 한 폴더에 모으려고 하면 SoT가 복제되어 stale 위험이 커진다.

### 산출물별 home

| 산출물                   | 위치                                                                                       |
| --------------------- | ---------------------------------------------------------------------------------------- |
| 사고·논의·옵션 비교           | GitHub Issue body + comments                                                             |
| 결정 (뒤집기 어려운 것)        | `docs/07-development/decisions/`                                                         |
| 재사용 가능한 카피·콘텐츠 SoT    | 영역별 SoT (`docs/01-profile/`, `docs/03-firm/`, `docs/02-services/`, `docs/04-cases/` 등)  |
| 페이지 전용 단발 카피          | 코드 안 (`src/app/(frontend)/.../page.tsx`)                                                 |
| 디자인 시안·와이어            | Figma 등 외부 + 캡처는 `docs/assets/`                                                          |

**원칙**: 페이지 단위로 산출물을 *미리* 모으는 폴더는 만들지 않는다. ADR에도 영역 SoT에도 안 맞는 페이지 전용 산출물이 2~3건 쌓이는 시점에 다음 구조로 lazy 신설하고, 신설 자체를 결정 이력에 기록한다.

```
docs/pages/
├── main/
├── about/
├── services/
└── cases/
```

페이지 단위 폴더가 생기더라도 *재사용 가능한 콘텐츠는 영역 SoT에 남기고*, 페이지 폴더에는 페이지 전용(섹션 구성·SEO 메타·페이지 한정 카피·와이어 메모)만 둔다.

**자주 헷갈리는 케이스**:
- 카피가 *이 페이지 전용*인지 *영역에서 재사용 가능*한지 애매할 때 → 일단 영역 SoT에. 한 곳에서만 쓰여도 SoT는 부담이 작지만, 코드에 박힌 카피를 SoT로 끌어올리는 건 고통스럽다.
- 와이어를 ASCII로 그려두고 싶을 때 → ADR이거나 issue comment. 별도 파일은 만들지 말 것.
- 인터뷰·자료조사 *원문*이 새로 들어올 때 → `docs/06-interview/` 또는 영역별 SoT (출처 보존 우선).

## 운영 룰

- **신규 결정**: `decisions/YYYY-MM-DD-<slug>.md` 형식으로 파일 추가, 위 표에 1줄 등록.
- **결정 뒤집기**: 기존 파일은 *지우지 않음*. 새 결정 파일 본문에 `supersedes 2026-MM-DD-...` 명시.
- **Task 단위**: 영역(epic)은 4개 고정, sub-task는 *지금 알고 있는 만큼만* 만든다. "기획 결정" 자체가 task가 될 수 있다 (`planning` 라벨).
