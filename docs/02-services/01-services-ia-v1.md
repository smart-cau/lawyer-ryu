## 2. 대분류

### 2.1 형사전문 (criminal)

- **페르소나**: 형사 사건 의뢰인(피의자 + 일부 피해자). 변호사 USP가 가장 강하게 작동하는 영역
- **USP 메시지**: *"검사 19년 + 사법연수원 35기 + 대한변협 등록 형사법 전문 변호사 → 수사 단계에서 무혐의·기소유예 종결"*
- **톤**: 신속·전문성·결과 중심. 시급성 강조 (24시간 연락, 지역=창원)

### 2.2 민사·가사·행정 (non-criminal)

- **페르소나**: 비형사 분쟁 의뢰인 (개인/가족/자영업자/공직자)
- **포지셔닝**: 형사 전문의 *보조 영역* — 형사와 연동되는 사건(음주운전 면허취소, 공무원 징계 등)에서 USP와 연결
- **톤**: 신뢰·이해 중심. 형사 대비 차분한 톤

## 3. 형사전문 leaf 상세

> **표 칼럼 정의** (§3·§4 모든 표 공통)
>
> - **라벨**: leaf hub 안 subcategory의 노출 라벨. 의뢰인 검색 키워드 직노출(§7) 기준.
> - **매핑 키워드**: 본 라벨에 포함되는 *세부 죄목·세부 형태·세부 서비스 항목*. SEO 키워드가 아니라 leaf가 다루는 **실제 범위 단위**. UI(예: services hub 카드 칩, leaf hub subcategory 카드)에서는 이 단위를 발췌해 노출.
> - **핵심 SEO 키워드**: 본 라벨로 페이지 메타·H2·H3에서 노출할 검색 키워드 표현 (별칭·통칭 포함, 예: "윤창호법", "김영란법").
> - **혐의 분류**: 본 라벨에 적용되는 관련 법령 근거. UI 노출 비-필수.

> **열린 결정 — 표 칼럼 구조 (2026-05-08 보류)**
>
> 아래 leaf 표들의 마지막 칼럼이 일관성 없음:
> - 3.1 / 3.3 / 3.4 / 3.5 / 3.7 / 3.8 → `핵심 SEO 키워드`
> - 3.2 / 3.6 → `혐의 분류` (관련 법령)
>
> **보류 이유**: 두 칼럼은 서로 다른 차원의 정보(SEO 키워드 vs 법령 근거)지만, leaf hub의 subcategory 카드 UI가 v1에서 어떻게 그려질지(키워드 칩 노출? 근거 법령 라인 노출?)가 미결정이라 칼럼의 역할을 확정할 수 없음. **칼럼이 카드 UI를 따라가야 함 (역방향 X)** — 카드 UI 없이 칼럼부터 표준화하면 추측 기반 채움이 됨.
>
> **결정 트리거**: leaf hub 페이지 템플릿 / subcategory 카드 UI 디자인 시점에 재검토.
>
> **후보 옵션**:
> - (A) 4칼럼 `라벨 | 매핑 키워드 | 핵심 SEO 키워드 | 관련 법령` — 카드에 키워드 칩 + 법령 라인 둘 다 노출 시
> - (B) 3칼럼 `라벨 | 매핑 키워드 | 핵심 SEO 키워드` — 카드에 키워드 칩만 노출 시
> - (C) 2칼럼 `라벨 | 매핑 키워드` + 표 아래 메모 블록(SEO 타겟·근거 법령) — 카드가 제목+1~2줄 설명만일 때
>
> 그때까지 기존 표 구조는 손대지 말 것 (정보 손실 위험). 새로 leaf를 추가할 때만 현 패턴(`라벨 | 매핑 키워드 | 핵심 SEO 키워드`) 따르기.

**시드 카테고리 설계 (3 layer · 56건)**


| Layer           | 개수  | 예시 (title / slug / parent)                                                                 |
| --------------- | --- | ------------------------------------------------------------------------------------------ |
| 1 (대분류)         | 2   | 형사전문 / `criminal` / null 민사·가사·행정 / `non-criminal` / null                                  |
| 2 (leaf 분야)     | 10  | 성범죄 / `sex-crime` / criminal 음주운전·교통범죄 / `traffic-crime` / criminal ...                    |

**slug 명명 규칙**: `<대분류 또는 leaf slug>-<영문 키워드>` — Posts 분류 필터링 시 prefix 매칭 가능.

**Posts 매핑**:

- Posts(성공사례)는 **subcategory(Layer 3)** 단위로 분류 (가장 세분화된 단위 매칭)
- 부모(leaf·대분류)는 자식 sub의 parent 참조로 자동 집계
- 하나의 Post가 여러 subcategory에 속할 수 있음 (m:n)

**적재 시점**: Posts(성공사례) 콘텐츠 수집 단계와 동기화. v1 IA 확정 시점엔 시드 설계만 본 문서에 보존, 실제 마이그레이션·시드 적재는 **별도 단계**.

**대안 (컬렉션 변경 부담이 클 때)**: 현재 컬렉션 그대로 + slug prefix만으로 계층 표현. Layer 3 카테고리만 적재. 부모 추적은 slug parsing.
