>
> ⚠ **F/P/S/M 코드 정의는 계약서 본문 내 별첨 또는 기획안 부록**에 정의되어 있을 가능성. 본 문서의 매핑은 합리적 추론이며, 계약서 원본 정의가 확인되면 정정한다.
>
> ⚠ **메인 섹션 수 변경**: 계약 원문은 "메인 8섹션"이나 변호사 결정으로 **언론·보도 섹션 제거** → 7섹션. 8 vs 7 차이는 추후 갑/을 협의 안건.

> ⚠ **분야 수 변경**: 계약 원본은 *"S-01 ~ S-N (갑 제안에 따른다)"* — 옵션 D는 6분야 제안.

> ⚠ **About 단일 페이지 결정**: P-01·P-02·P-03 분리 → **P-01 단일 long-form 통합**. USP 키워드 4종(검사출신·부부장·류남경·경남) 모두 ≤30/월로 페이지 분리 정당성 부족. 17 → 12단위 변경 (-29%)에 따른 페이지 단가/일정 영향은 갑/을 협의 안건.

## 4. M-01 ~ M-04 — 성공사례 + CMS

> 계약서 원문: *"성공사례 페이지 및 성공사례 관리 시스템(CMS)"*
> 4개 단위는 "프론트(고객 노출) 2 + 어드민(CMS) 2" 또는 "프론트 1 + 어드민 3" 패턴이 일반적.

### 권장 매핑 A — 프론트 2 + CMS 2

| 코드 | 영역 (추론) | URL / 경로 | 핵심 기능 |
|:---:|------|-----------|-----------|
| **M-01** | 성공사례 목록 (프론트) | `/cases/` | 카드 그리드, 필터(분야), 페이지네이션 |
| **M-02** | 성공사례 상세 (프론트) | `/cases/[slug]/` | 사건 개요·결과·후기, BreadcrumbList JSON-LD |
| **M-03** | CMS — 사례 작성/편집 | `/admin/cases/` | 인증 후 사례 CRUD, 분야 태그·게시 토글 |
| **M-04** | CMS — 분류·인증 관리 | `/admin/settings/` 또는 `/admin/login/` | 카테고리 관리 + 어드민 로그인 |

### 권장 매핑 B — 프론트 1 + CMS 3

| 코드 | 영역 | URL / 경로 |
|:---:|------|-----------|
| **M-01** | 성공사례 (프론트, 목록+상세 통합) | `/cases/` (모달 또는 expandable) |
| **M-02** | CMS — 로그인/인증 | `/admin/login/` |
| **M-03** | CMS — 사례 CRUD | `/admin/cases/` |
| **M-04** | CMS — 분류 / 메타 관리 | `/admin/categories/` |

> ⚠ A vs B 결정은 계약서 본문 정의 또는 디자인 미팅 시 확정.
> 일반적으로 **A안(프론트 분리)**이 SEO·UX 측면에서 유리.

### CMS 최소 요건 (계약 추정)
- 인증된 어드민 1~2명 (대표변호사 본인 + 사무직원)
- 사례 필드: 제목, 분야, 사건 개요, 결과, 게시일, 노출/숨김, 대표 이미지(선택)
- 분야 매핑: S-01·S-02·S-03·S-05·S-08·S-09 (옵션 D 6개)와 동일 카테고리 사용
- 게시 후 프론트 즉시 반영 (ISR 또는 on-demand revalidation)
