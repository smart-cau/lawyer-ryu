# Wireframe Process

이 프로젝트의 "wireframe"은 일반적인 low-fi wireframe(흑백 박스의 빠른 발산)이 아니라 **mid-fi UX validation prototype**이다. 디자인 토큰을 적용한 *실제 코드*로 페이지를 만들고, 그걸 *읽어보면서* 의도한 사용자 경험이 작동하는지 검증한다. 코드를 매체로 쓰는 이유는 매체 자체가 검증 도구이기 때문이다 — Figma·종이로는 페이지를 "읽어볼" 수 없다.

신규 페이지·신규 섹션 작업 진입 시 이 skill의 체크리스트를 따른다.

---

## 이 단계의 목적과 종료 조건

### 목적

코드 산출물 위에서 다음 세 가지를 확인한다.

- 메시지·콘텐츠·정보 위계가 *읽혔는가*
- 페이지 흐름과 컴포넌트 간 연결이 *의도대로 움직이는가*
- 다음 단계로 넘어갈 때 남는 일이 **배치 디테일·색·카피 문구의 finishing**으로 한정되는가

### 종료 조건

위 세 가지가 통과하면 wireframe 단계 종료. 종료 후 남는 작업은 *finishing 단계*다.

- 색·브랜드 표현의 정교화
- 카피의 글자 단위 다듬기 (자료조사 메모 참고해 사용자 확인 후 확정)
- 타이포·spacing·shadow·hover의 미세조정
- 마이크로인터랙션·접근성 디테일

wireframe 단계에서는 finishing 항목으로 정의된 것들을 미룬다.

### 코드는 검증 *전*이 아니라 *과정*으로 쓴다

이 단계는 "코드 작성 전에 합의를 끝내고, 코드는 합의의 산출물"이라는 직선 흐름이 아니다.

```
합의 → 1차 코드 → 시각 검증·정정(1~2회) → 종료
```

이게 정상 루프다. 합의 안에서 *텍스트만으로 결정 가능한 항목*과 *시각 결과물을 봐야 결정되는 항목*을 구분한다 — 아래 "가설 vs. 짐작" 섹션이 이 구분의 룰.

## 검증하는 것 — 우선순위 순

### 1. 메시지 (가장 먼저, 가장 자주 놓치는 것)

- 이 섹션이 의뢰인에게 남기는 **단 하나의 takeaway**는 무엇인가
- 스타일·이미지를 모두 벗겨도 그 메시지가 남는가
- 인접 섹션과 takeaway가 **의식적으로 다른가** (중복되면 한 섹션은 군더더기)

> 예: 메인 USP(M-02)는 "왜 가능한가/원칙", Process(M-05)는 "어떻게 진행되는가/행동" — 둘이 같은 메시지가 되지 않도록 layer 분리. (`docs/03-firm/main-ia-v0.1.md` 참고)

### 2. Content inventory (배치보다 선행)

- 이 섹션에 들어갈 **실제 콘텐츠의 양과 종류**가 확정되었나
- 카드 4장? 6장? 3장 + 1장 강조?
- 이미지를 **배치할지 말지**가 의식적으로 결정됐나. 배치한다면 몇 장? 비율(16:9·1:1·4:3 등)은? grid인지 단일인지?
- 카피 후보가 결정됐나, 아니면 카피 단계에서 다시 흔들릴 여지가 있나
- 각 카피 슬롯의 **종류**가 명시됐나 (eyebrow / title / subtitle / lead / body / caption / CTA label 등)

> 콘텐츠 양이 미정인 채로 배치를 잡으면 무조건 재작업. **카피 후보를 `docs/` 자료조사 메모에서 참고해 사용자와 함께 확정한 후** layout 결정으로 넘어간다. `docs/`는 참고 자료이지 발췌해서 그대로 박는 원천이 아니다.

#### 카피 *문구*는 미뤄도 되지만, 카피 *종류*는 미루지 않는다

글자 단위 문구는 카피 단계에서 다듬으면 된다. 하지만 그 슬롯이 **어떤 역할의 카피인지**는 wireframe에서 확정한다. 카피 종류가 배치를 좌우하기 때문이다.

- `eyebrow + h2 + lead` 패턴과 `h2 + body` 패턴은 시각 위계·spacing·typography 토큰이 다르게 적용된다
- 카드 한 장이 `title + body`인지 `number + title + caption`인지에 따라 grid 자식 요소 비율(5-b)이 바뀐다
- "여기에 뭔가 짧은 글이 들어간다"는 합의가 아니다 — 그 글이 *title*인지 *caption*인지에 따라 사이즈·정렬·여백이 전부 달라진다

> ❌ "카피는 추후 결정, 일단 한 줄짜리 placeholder" — 종류 미정 상태
> ✅ "이 슬롯은 lead (1~2문장), 그 아래 body (3~4문장)" — 종류 확정, 문구는 카피 단계로 미룸 (자료조사 메모를 참고해 사용자 확인을 거쳐 확정)

### 3. 정보 위계 (Hierarchy)

- 의뢰인의 시선이 1순위 → 2순위 → 3순위로 자연스럽게 흐르는가
- "가장 중요한 게 가장 두드러진다" 원칙 충족하는가
- 균질 노출 vs. 강조 분리: 의도적인 결정인가

> 예: USP 4카드를 균질하게 보이게 할지, A·B만 강조하고 C·E를 보조로 처리할지는 의식적 결정. "그냥 grid에 4장"은 결정이 아니라 회피.

### 4. 흐름과 연결 — 섹션 간 + 페이지 간

#### 4-a. 섹션 간 리듬 / 일관성

- 이 섹션을 인접 섹션 옆에 두면 사용자가 "같은 사이트"로 느끼는가
- 섹션 길이가 비슷한가, 의도적으로 다른가
- 같은 시각 패턴(예: overline + h2 + lead)을 반복하는가, 의도적으로 변주하는가
- Section rhythm 토큰(`py-section`, `space-y-section`)을 일관 적용하는가 — `.claude/rules/design-system.md`

#### 4-b. 페이지 간 흐름 (cross-page)

- 이 섹션이 다른 페이지로 향하는 진입점인가 (예: services hero → leaf, hero CTA → 연락)
- 사용자가 다음 행동으로 갈 합리적 경로가 *시각적으로* 보이는가
- 진입점의 단서(아이콘·라벨·CTA 문구)와 도착 페이지의 약속이 일치하는가

> 메인 페이지 작업 중에는 단일 페이지지만, services hub → leaf, hero → 연락 등 페이지 간 연결이 시작되면 이 항목이 즉시 활성화된다.

### 5. 배치 (Layout)

1~4가 정해진 *결과*로 배치가 도출된다. 7개 항목을 *모두 합의*하지만, 합의 방식이 **사전 합의 필수**와 **1차 가설 + 시각 정정**으로 나뉜다.

#### 5-A. 사전 합의 필수 — 시각 정정 비용이 매우 큰 항목

코드로 만든 후 정정하면 구조를 통째로 재작성해야 한다. 텍스트 합의로 끝장을 본다.

| # | 항목 | 결정해야 하는 것 |
|---|---|---|
| 5-a | **Layout 형식** | card grid / editorial list / asymmetric zigzag / statement-quote / 기타. 메시지(1단계)에서 도출. `feedback_layout_decisions.md` 참조 — IA·스텁의 "N장 카드" 표현을 layout 약속으로 받지 말 것 |
| 5-b | **Grid 구조** | col 수, gap, 자식 요소 비율 (예: 12-col 중 number col-2 / content col-10) |
| 5-g | **모바일 collapse 동작** | desktop grid가 mobile에서 단일 컬럼으로 어떻게 변하는지. 번호·라벨 등 보조 요소가 어디로 이동하는지 |

#### 5-B. 1차 가설 + 시각 정정 — 시각으로 봐야 판단되는 항목

텍스트 합의로 결정해도 시각 결과물을 보면 어차피 흔들린다. **1차 가설을 명시**하고 코드로 만들어 함께 본 다음 1~2회 iterate한다. 가설임을 합의 텍스트에 명시하는 한 짐작 금지 룰 위반이 아니다.

| # | 항목 | 1차 가설 형식 예 |
|---|---|---|
| 5-c | **콘텐츠 max-width** | "1차 가설: `max-w-5xl`. 시각 검토 후 좁힐지 결정" |
| 5-d | **헤더 정렬** | "1차 가설: `text-center + mx-auto`. 좌측이 나으면 정정" |
| 5-e | **본문 정렬** | "1차 가설: `text-left`. 헤더와 일치/대비 검토" |
| 5-f | **헤더-본문 시각적 관계** | "1차 가설: 헤더 `max-w-3xl` 가운데 + 본문 `max-w-5xl` 가운데. 시작점 비대칭 의도" |

#### 합의 형식

- 5-A 3개는 *완전 합의* 상태로 코드 진입
- 5-B 4개는 *1차 가설로 명시*하고 코드 진입 — "이 4개는 1차 가설이며 결과물 위에서 정정한다"는 합의 자체를 텍스트에 남긴다
- ASCII 스케치는 보조 시각자료. ASCII만으로 충분하지 않다 — 위 7개 항목을 체크리스트로 빠짐없이 적시한다

#### 정정 루프 종료

5-B 항목들이 시각 검토에서 더 이상 정정 의견이 없으면 *확정*으로 승격. 이 시점 이후 같은 항목을 또 흔드는 건 finishing 단계로 미룬다.

### 6. Affordance (interactive 식별)

- 카드가 클릭 가능한가, 정적인가
- CTA 위치·유무·문구
- 호버/포커스에 의미가 있는가, 단순 장식인가

### 7. UX 검증 통과 기준

코드 산출물 위에서 *읽어보며* 다음을 확인한다. 5초 테스트는 이 체크리스트의 한 항목이다.

#### 가정 명시

- 어떤 의뢰인 persona가 (예: 형사사건 처음 겪는 사람, 비형사 사건 위임 검토자)
- 어디서 들어와서 (예: 네이버 검색, 블로그 링크, 명함 QR)
- 무엇을 하려고 하는가 (예: 상담 가능성 판단, 변호사 신뢰도 확인, 전화번호 확보)

가정이 적히지 않으면 검증 기준이 떠다닌다.

#### 통과 기준

- **5초 테스트**: 섹션 첫 화면에서 5초 안에 메시지가 잡히는가
- **다음 행동 명료성**: 스크롤 후·페이지 끝에서 "다음 행동"이 명확한가 (CTA·다른 페이지로의 진입)
- **경로 합리성**: persona의 task가 합리적 클릭/스크롤 횟수로 달성 가능한가
- **중복·군더더기 없음**: 인접 섹션·페이지의 메시지가 중복되지 않는가
- **모바일도 같이 통과**: desktop만 통과하고 mobile에서 떨어지는 항목이 없는가

#### 불통과 시

- 어떤 기준에서 떨어졌는지 명시 (예: "5초 테스트는 통과, 다음 행동 모호")
- 원인이 *어느 체크리스트로* 거슬러 올라가는지 정렬 (메시지 1번 / 위계 3번 / 흐름 4번 등)
- 색·shadow·디테일을 더해서 해결되는 문제가 아니라면 구조로 돌아간다

## 가설 vs. 짐작 — 정정 루프의 정의

이 단계의 흐름은 "코드 작성 전에 모든 결정을 끝낸다"가 아니라 "**합의의 종류에 따라 결정을 코드 전/후로 나눈다**"이다.

### 짐작 — 금지

합의되지 않은 항목을 *결정한 척* 코드로 만드는 것. 합의 텍스트에 "1차 가설"이라고 명시하지 않은 채 기본값으로 채우는 것.

- ❌ "헤더 정렬은 안 정해졌지만 좌측이 default니까"
- ❌ "일단 좌측 정렬로 가고 나중에 정정" — 합의 텍스트에 가설로 박혀 있지 않음
- ❌ "max-width 명시 안 됐지만 보통 container 전체로"

### 가설 — 정상

합의된 *1차 가설*을 명시하고 코드로 만든 후 시각 결과물 위에서 정정하는 것. **합의 텍스트에 "1차 가설" 또는 "코드 후 결정"으로 박혀 있다는 점이 짐작과의 유일한 차이.**

- ✅ "5-d 헤더 정렬: 1차 가설 `text-center + mx-auto`. 결과물 보고 좌측 정렬과 비교 후 결정"
- ✅ "5-c max-width: 1차 가설 `max-w-5xl`. 좁혔을 때 본문 줄바꿈 어색해지면 `max-w-4xl`로 정정"

### 가설은 5-B 항목에만 허용

5-A(layout 형식·grid 구조·모바일 collapse)에는 가설 형식이 허용되지 않는다 — 시각 정정 비용이 *구조 재작성* 수준이라 텍스트 합의 단계에서 확정해야 한다.

### 사례

> 2026-05-11 USP 섹션 wireframe. 5-c (max-width), 5-d (헤더 정렬)를 합의 텍스트에 *가설로도 명시하지 않고* 기본값으로 코드 작성 → 사용자 피드백으로 2회 정정. 만약 합의 시 "5-c, 5-d는 1차 가설로 코드 → 시각 정정"으로 명시했다면 같은 2회 정정도 정상 루프였을 일.

## 검증하지 않는 것 — finishing 단계로 미루는 것

wireframe 단계에서 신경 쓰지 않는다. 종료 후 finishing에서 다룬다.

- 색·브랜드 표현 (디자인 토큰이 이미 정해져 있으면 그것만 적용)
- 폰트 weight/size 미세조정 (semantic 토큰으로 일단 적용, 미세조정은 별도 단계)
- shadow·border-radius·hover 디테일
- 마이크로인터랙션·전환 디테일
- 이미지의 최종 비주얼·종류·구체적 콘텐츠 (placeholder로 충분)
  - **단, 이미지 *배치 여부*와 *비율*은 검증 대상이다** — 이미지가 있고 없고에 따라 grid 구조(5-b)와 콘텐츠 max-width(5-c)가 통째로 바뀐다. Content inventory(체크리스트 2번)에서 합의.
- 카피의 글자 단위 다듬기 (의미만 맞으면 됨)
  - **단, 카피의 *종류*(eyebrow/title/sub/lead/body/caption 등)는 검증 대상이다** — Content inventory(체크리스트 2번) 참조. 종류가 배치를 좌우하므로 layout 결정 전에 합의한다.

## 섹션 단위 진행 사이클

각 섹션마다 동일한 7단계를 진행한다.

```
1) 메시지               — 이 섹션이 남기는 단 하나의 문장
2) Content              — 콘텐츠 수/구조, 카피 종류, 이미지 배치 여부·비율
3) 위계                 — 무엇이 1순위로 보여야 하는가
4) 흐름·연결            — 섹션 간(4-a) + 페이지 간(4-b)
5) 배치
   5-A) 사전 합의 필수  — Layout 형식 / Grid 구조 / 모바일 collapse
   5-B) 1차 가설        — max-width / 헤더 정렬 / 본문 정렬 / 시각 관계
6) Affordance           — 클릭 가능한 요소·CTA
7) UX 검증              — 코드 결과물 위에서 통과 기준 적용
```

### 순차 진행 원칙

여러 섹션을 한 번에 wireframe하지 않는다. **섹션 단위로 합의 → 1차 구현 → 시각 정정 → commit → 다음 섹션**.

이유:
- 인접 섹션 관계(4-a)는 앞 섹션이 *확정*된 후에만 검증 가능
- 코드 매체에서는 섹션 단위 commit이 변경 단위로 자연스러움

#### 비용 인정 (트레이드오프)

일반 wireframe best practice는 "페이지 전체를 한 번에 보고 균형 검증" 방향이라 우리와 정반대다. 우리 선택의 비용:

- 페이지 전체 리듬은 N번째 섹션이 끝나야 보임. N-1 시점의 결정이 N+2에서 흔들릴 수 있음
- 이 비용을 줄이기 위해 page IA(`docs/<area>/...-ia-...md`)를 *섹션 작업 전에* 합의해 둔다 — IA가 페이지 전체 균형의 사전 합의 매체

### 예외

단일 섹션이 매우 단순하고(예: Contact 정보 4줄) 위계 결정 여지가 거의 없을 때는 합의를 짧게 압축해도 된다. *생략*은 아니고 *압축*.

## 부가 best practice

### 모바일을 먼저 그려본다

데스크톱 4-col 카드는 모바일에서 4-row 타워. 모바일에서 자연스러운지 먼저 보는 게 효율적. 실제 구현은 Tailwind mobile-first 컨벤션 그대로(`grid-cols-1 lg:grid-cols-4`).

### 카피와 layout을 동시에 결정하지 않는다

카피 길이가 흔들리면 layout도 흔들린다. **카피 후보를 먼저 확정 → layout은 그 카피에 맞춰서** 잡는다. 거꾸로 하면 카피가 layout에 끌려가서 의도와 어긋난다. 카피 후보는 `docs/` 자료조사 메모를 참고하되 *사용자 확인을 거친 후* 확정한다.

### 카피는 자료조사 메모를 참고하되 사용자 확인 후 확정

Wireframe 단계에서 새 카피를 *임의로* 쓰지 않는다. `docs/03-firm/`, `docs/02-services/`, `docs/01-profile/` 등 자료조사 메모를 *참고*해 후보를 제시하고, **사용자와 합의해 확정**한다. `docs/`는 자료조사 수준이라 그대로 발췌해 박을 권위가 없다 — 문구·표현은 합의 단계에서 다듬는다. 자료조사에 후보가 없다면 카피 단계의 일이지 wireframe의 일이 아니다.

### Placeholder의 충실도

- 텍스트: 실제 카피 후보 (자료조사 메모 참고 + 사용자 확인). lorem ipsum 금지 — 길이감을 못 봄
- 이미지: **URL placeholder 서비스로 dimension만 표현**한다. gray div를 직접 그리지 말 것 — 비율 변경이 느리고, "최종 결정 아님" 신호가 약하다
  - 추천: `https://placehold.co/<W>x<H>?text=<라벨>` (예: `https://placehold.co/1600x900?text=Hero+Image`). 라벨에 슬롯의 *역할*(Hero / Portrait / Case thumbnail 등)을 박는다 — 콘텐츠가 아니라 자리만 표시
  - `picsum.photos` 같은 실사진 서비스는 mockup 느낌이라 wireframe 의도(`결정 안 됨` 시각화)와 어긋난다 — 피한다
  - 실제 이미지가 `public/`에 이미 들어와 있으면 그걸 써도 되지만, **"최종 결정된 visual 아님"을 합의 텍스트에 명시**한다
  - **Next.js 주의**: wireframe 단계에서는 `next/image`보다 `<img>` 태그를 쓴다. `next/image`로 외부 placeholder URL을 쓰려면 `next.config.ts`의 `images.remotePatterns`에 도메인 등록이 필요해서 마찰이 생긴다
- 카드: shadow·radius는 디자인 토큰 기본값만, hover 무시

## 산출물·문서화

### Wireframe 자체는 코드로 남긴다

이 프로젝트는 `_components/` 안에 wireframe-level 컴포넌트를 두는 방식으로 진행한다. 별도 와이어 파일(Figma·ASCII 파일)을 만들지 않는다.

이유: 코드 자체가 단일 진실 원천이 되어야 stale 위험이 없다. `docs/07-development/INDEX.md`의 "와이어를 ASCII로 그려두고 싶을 때 → 별도 파일은 만들지 말 것" 원칙과 정합.

### 합의 과정은 conversation에 남긴다

체크리스트 1~7 합의는 대화에서 진행한다. 결정 자체가 비자명하거나 뒤집기 어려운 경우에만 ADR(`docs/07-development/decisions/`) 또는 영역 자료조사 메모 업데이트로 영구화한다.

### 페이지 IA가 흔들리면 IA 문서 먼저 갱신

섹션 wireframe 진행 중에 IA 자체의 변경 필요가 발견되면(예: M-04와 M-05 순서 교체, 슬롯 신설), 코드 변경 *전에* 해당 IA 문서(`docs/<area>/...-ia-...md`)부터 갱신하고 합의한다. 코드와 자료조사 메모가 어긋나면 추적이 깨진다.

## 정의·수정 원칙

- 이 skill을 수정해야 한다고 느낀 변경(체크리스트 추가, 사이클 변경, 예외 케이스 발견)은 **반드시 이 파일도 함께 업데이트**한다
- 단발성 결정(예: "이번 섹션만 7단계 압축")은 이 문서에 기록하지 않는다 — 대화에 남기는 걸로 충분
- 이 룰을 깨야 할 명확한 이유가 생기면 깨도 된다. 단, *왜 깼는지*를 대화 또는 ADR에 남긴다.
